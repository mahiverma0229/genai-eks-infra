# Production-Grade GenAI Inference Platform on Kubernetes

## Overview

This project demonstrates how to deploy and operate an AI inference application on Kubernetes using production-grade DevOps and SRE practices.

The primary goal of this project was **not to build or train an LLM**, but to build a reliable Kubernetes platform capable of safely deploying and operating AI workloads.

The project uses a lightweight **TinyLlama** model for inference while implementing modern platform engineering concepts such as:

- Kubernetes
- Canary Deployments
- GitOps
- Observability
- Horizontal Pod Autoscaling
- SLO-based Rollouts
- Self-Healing

---

# Architecture

Client requests are routed through an Ingress controller to a Kubernetes Service.

The Service distributes traffic across multiple TinyLlama inference pods.

Prometheus continuously collects application and infrastructure metrics.

Grafana visualizes these metrics for monitoring.

Argo Rollouts manages canary deployments and gradually shifts traffic to new application versions while monitoring service health.

# Features

## 1. LLM Inference Service

The application exposes a REST API that accepts prompts from users and returns responses generated by the TinyLlama model.

Example:

POST /generate

```json
{
    "prompt": "Explain Kubernetes."
}
```

Response

```json
{
    "response": "Kubernetes is a container orchestration platform..."
}
```

---

## 2. Canary Deployments

Instead of deploying a new version to all users immediately, Argo Rollouts gradually shifts traffic.

Example rollout:

- 10% traffic
- 25% traffic
- 50% traffic
- 100% traffic

During each step, application health is monitored before moving to the next stage.

This reduces deployment risk and limits the impact of faulty releases.

---

## 3. Health Checks

Each inference pod includes:

### Readiness Probe

Checks whether the application is ready to receive requests.

If the readiness probe fails, Kubernetes removes the pod from the Service until it becomes healthy.

### Liveness Probe

Checks whether the application is still running correctly.

If the application becomes unresponsive, Kubernetes automatically restarts the pod.

---

## 4. Horizontal Pod Autoscaling

Horizontal Pod Autoscaler automatically scales the inference pods based on CPU utilization.

Configuration:

- Minimum replicas: 2
- Maximum replicas: 10
- Target CPU: 70%

This allows the platform to handle increased traffic without manual intervention.

---

## 5. Observability

Prometheus collects metrics from:

- Kubernetes Nodes
- Pods
- TinyLlama inference service

Collected metrics include:

- Request Rate
- Request Latency
- Error Rate
- CPU Usage
- Memory Usage
- Pod Restarts

Grafana dashboards visualize these metrics in real time.

---

## 6. SLI / SLO Monitoring

The platform defines Service Level Indicators such as:

- Request Latency
- Availability
- Error Rate

Example Service Level Objective:

- 95% of requests should complete within 500ms.

These SLOs are monitored continuously during deployments.

---

## 7. Safe Rollouts

Argo Rollouts monitors Prometheus metrics during every deployment.

If latency or error rate increases beyond the acceptable threshold, the rollout can be paused or rolled back.

This ensures that only healthy versions are promoted.

# End-to-End Request Flow

1. User sends a request to the inference API.
2. The Ingress receives the request.
3. The Kubernetes Service forwards it to one of the TinyLlama inference pods.
4. The pod processes the prompt and generates a response.
5. Prometheus collects application metrics.
6. Grafana displays dashboards for monitoring.
7. During new releases, Argo Rollouts gradually shifts traffic while monitoring application health.
